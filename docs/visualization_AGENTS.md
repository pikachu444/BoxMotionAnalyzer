# AGENTS.md: AI 에이전트를 위한 개발 가이드

안녕하세요, 동료 AI 에이전트 여러분! 이 문서는 여러분이 이 프로젝트를 더 빠르고 정확하게 이해하고, 효율적으로 작업할 수 있도록 돕기 위해 작성되었습니다. 작업을 시작하기 전에 반드시 이 문서를 정독해주세요.

## 1. 최우선 원칙: 테스트 기반 검증

**이 환경에서는 GUI를 직접 실행하여 테스트할 수 없습니다.** `python main.py`와 같은 명령어는 항상 실패합니다. 따라서, 코드 수정 후에는 반드시 **단위/통합 테스트 코드를 작성하여 기능을 검증**해야 합니다.

- **환경 제약:**
  - **헤드리스 환경:** GUI 디스플레이 서버(X-Server)가 없어 시각적인 UI를 렌더링할 수 없습니다.
  - **OpenGL 제한:** `pyvista`와 같은 3D 라이브러리는 `offscreen` 플랫폼 플러그인을 사용해도 OpenGL 기능 부재로 초기화에 실패할 수 있습니다. `VistaWidget`과 같이 렌더러를 직접 생성하는 클래스의 인스턴스화는 피해야 합니다.

- **권장 테스트 전략:**
  - **로직 테스트:** GUI 위젯을 직접 생성하는 대신, 위젯 내부의 핵심 데이터 처리 로직(예: `_get_points_for_ids`)을 복제하여 테스트하는 것이 효과적입니다.
  - **`QT_QPA_PLATFORM=offscreen`:** `QApplication` 인스턴스가 필요한 테스트의 경우, 명령어 앞에 이 환경 변수를 붙여주면 GUI 렌더링 시도 없이 테스트를 실행할 수 있습니다.

## 2. 핵심 아키텍처 및 데이터 흐름

이 프로젝트의 데이터 흐름은 매우 중요하며, 모든 작업은 이 흐름에 대한 이해를 바탕으로 이루어져야 합니다.

```
[1. config.py]
   - 모든 상수와 설정을 정의합니다.
     |
     v
[2. make_testdata.py]
   - config.py 설정을 기반으로 3-레벨 멀티헤더 CSV 파일을 생성합니다.
     |
     v
[3. Multi-header CSV (data/testdata_box_marker.csv)]
   - 모든 시계열 데이터의 원본 소스입니다.
     |
     v
[4. data_handler.py]
   - CSV 파일을 읽고, 모든 데이터를 담고 있는 표준화된 long-format Pandas DataFrame을 생성합니다.
   - 이 모듈은 데이터에 접근하는 유일한 통로(Single Source of Truth) 역할을 합니다.
     |
     v
[5. main_window.py]
   - 사용자의 입력(파일 열기, 재생, 객체 선택 등)을 받아 data_handler에 데이터를 요청하는 중앙 컨트롤러 역할을 합니다.
     |
     v
[6. vista_widget.py / plot_widget.py]
   - main_window를 통해 전달받은 데이터를 사용하여 3D 시각화 또는 2D 그래프를 그리는 최종 소비자입니다.
```

## 3. 데이터 구조의 역사와 현재

**과거 (Legacy):** 초기 버전은 **Wide-format CSV**를 사용했습니다. 각 객체의 각 축이 별도의 컬럼(예: `C0_x`, `C0_vel_x`)을 가졌습니다.
- **문제점:** 이 방식은 `data_handler.py`에서 복잡하고 오류가 발생하기 쉬운 정규식 파싱과 피벗 로직을 필요로 했습니다. 새로운 데이터 타입(예: `acceleration`) 추가가 매우 어려웠습니다.

**현재 (Current):** **3-레벨 멀티헤더 CSV**를 사용합니다.
- **구조:** 헤더가 `(data_type, object_id, axis)`의 3개 레벨로 구성됩니다.
  - `data_type`: `position`, `velocity` 등
  - `object_id`: `C0`, `MK_TOP_1` 등
  - `axis`: `x`, `y`, `z`
- **장점:** 데이터 구조가 명확하며, `pandas`의 표준 기능을 사용하여 훨씬 간결하고 안정적으로 데이터를 읽고 처리할 수 있습니다. 확장성이 매우 뛰어납니다.
- **주의:** 모든 데이터 처리 로직은 이 멀티헤더 구조를 전제로 작성되었습니다. 데이터 I/O 관련 수정 시에는 이 점을 반드시 인지해야 합니다.

## 4. 주요 모듈 가이드

- **`config.py`**
  - **역할:** "설정의 중앙 허브". 모든 상수, 경로, 라벨, 스타일 설정을 담고 있습니다.
  - **작업 지침:** 새로운 상수나 설정이 필요하면 반드시 이 파일에 추가하고, 다른 파일에서는 이 상수를 임포트해서 사용하세요. **하드코딩은 절대 금물입니다.** 특히 멀티헤더의 레벨 이름과 값 (`MH_*` 상수)을 반드시 사용해야 합니다.

- **`data_handler.py`**
  - **역할:** "데이터의 문지기". CSV 파일 I/O 및 DataFrame 변환을 독점적으로 책임집니다.
  - **작업 지침:** 데이터 구조 변경이나 처리 로직 수정이 필요할 경우, 이 파일이 가장 먼저 수정되어야 합니다. 외부 API(`get_frame_data`, `get_object_timeseries` 등)는 최대한 일관성을 유지하여 다른 모듈에 미치는 영향을 최소화해야 합니다.

- **`vista_widget.py`**
  - **역할:** "3D 렌더링 엔진". PyVista를 사용하여 3D 객체를 화면에 표시합니다.
  - **작업 지침:**
    - **라이브러리 사용 주의:** PyVista의 함수나 속성을 사용할 때는, 과거에 존재하지 않는 함수(`plotter.camera.position` 등)를 사용하여 오류를 일으킨 사례가 있습니다. **기능을 추가하거나 수정할 때는 반드시 PyVista 공식 문서를 참조하여 정확한 API를 사용해야 합니다.**
    - **성능:** `update_view` 함수는 매 프레임마다 호출되므로, 이 함수 내의 로직은 최대한 효율적으로 작성되어야 합니다. 불필요한 객체 재생성을 피하고, 기존 `PolyData`의 `points`를 업데이트하는 방식을 사용하세요.


## 5. 향후 개발 방향 및 설계 결정

다음 기능들을 개발할 때 아래의 가이드라인을 따라주세요.

-   **그래프 상세 분석 기능 (팝업, 툴바, 값 표시):**
    -   **트리거:** `PlotWidget`에 대한 `mouseDoubleClickEvent`를 감지하여 팝업을 실행한다.
    -   **팝업 구현:** `QDialog`를 사용하고, 그 안에 `QVBoxLayout`을 배치한다.
    -   **내용물:** 새로운 `PlotWidget` 인스턴스와 `matplotlib.backends.backend_qt5agg.NavigationToolbar2QT` 툴바를 레이아웃에 추가한다.
    -   **값 표시:** `motion_notify_event`를 `mpl_connect`하여 마우스 위치에 가장 가까운 데이터 포인트를 찾고, `ax.annotate`를 사용하여 툴팁을 구현한다. 이전 툴팁은 다음 이벤트 발생 시 반드시 `remove()` 해야 한다.

-   **특정 프레임 구간 플로팅:**
    -   **UI:** 복잡한 커스텀 슬라이더 대신, **`QCheckBox`와 `QSpinBox` 2개**를 사용하는 "체크 후 활성화" 방식을 채택한다. 이는 의존성을 추가하지 않고, 사용자에게 명확한 UX를 제공한다.
    -   **로직:** `main_window.py`에서 체크박스 상태와 스핀박스 값을 확인하여, `data_handler`에서 받은 데이터를 필터링한 후 `plot_widget`에 넘겨준다.

-   **동적 정보 로그:**
    -   **UI:** `QTextEdit` 대신 **`QTableWidget`**을 사용한다.
    -   **구조:** 선택된 객체(`object_id`)를 테이블의 **컬럼**으로, 사용자가 체크한 정보(`Position`, `Speed` 등)를 테이블의 **행**으로 하는 동적 구조를 채택한다.
    -   **데이터 처리:**
        -   `Speed`와 같이 원본 데이터에 없는 값은 `np.linalg.norm` 등을 사용하여 동적으로 계산한다.
        -   특정 객체에 해당 속성이 없는 경우(예: 마커의 회전 속도), 해당 테이블 셀에 **"N/A"** 문자열을 표시하여 데이터가 없음을 명확히 알려준다.
        -   UI를 분리하지 않고 하나의 테이블에서 모든 정보를 처리하여 일관성을 유지한다.

-   **설계 패턴: 단일 책임 원칙 (SRP) 적용**
    -   **문제:** `ControlPanel`이 제어 위젯(버튼, 리스트)과 정보 표시 위젯(`Information Log`)을 모두 관리하여 두 가지 책임을 가지고 있었다.
    -   **해결:** `Information Log` 관련 UI와 로직을 `info_log_widget.py`의 `InfoLogWidget` 클래스로 완전히 분리했다. `MainWindow`가 각 독립적인 위젯(`ControlPanel`, `PlotWidget`, `InfoLogWidget`)을 조립하는 책임을 맡는다.
    -   **장점:** 각 클래스의 역할이 명확해지고, 코드의 응집도가 높아졌으며, 다른 부분에 영향을 주지 않고 특정 위젯을 수정하거나 교체하기 쉬워졌다.

-   **설계 패턴: 상태 기반 업데이트와 범용 슬롯**
    -   **문제:** `update_plot_with_multiple_objects`와 같이 여러 다른 시그널에 연결된 슬롯이, 각 시그널이 전달하는 인자의 타입(예: `list` vs `int`)이 달라 `TypeError`를 유발하는 경우가 있었다.
    -   **해결:** **슬롯(함수)이 인자를 받지 않도록** 수정하고, 대신 함수 내부에서 `self.control_panel.object_list.selectedItems()`와 같이 **필요한 위젯의 현재 상태를 직접 참조**하도록 한다.
    -   **장점:** 이 "상태 기반 업데이트" 방식은 슬롯의 로직을 특정 시그널의 구현으로부터 분리(Decoupling)시킨다. 슬롯은 "어떤 이벤트가 발생했으니, 현재 UI 상태를 총 집합하여 화면을 갱신한다"는 단일 책임을 갖게 되어 코드가 더 견고하고 확장성이 높아진다.

-   **설계 패턴: 순환 참조 해결**
    -   **문제:** `PlotWidget`(그래프)과 `PlotDialog`(팝업)가 서로를 직접 임포트하여 순환 참조 문제가 발생했었다. '지역 임포트'는 임시방편일 뿐 좋은 해결책이 아니다.
    -   **해결:** **시그널-슬롯을 이용한 중재자 패턴**을 적용하여 해결했다.
        1.  `PlotWidget`은 더블클릭 시 `doubleClicked`라는 **시그널만 발생**시킨다. (`PlotDialog`의 존재를 모름)
        2.  `MainWindow`(중재자)가 이 시그널을 받아, `PlotDialog`를 **대신 생성하고 실행**한다.
        3.  이를 통해 각 위젯은 독립성을 유지하고, 상위 창이 상호작용을 제어하게 되어 더 깔끔하고 유지보수하기 좋은 구조가 되었다.
