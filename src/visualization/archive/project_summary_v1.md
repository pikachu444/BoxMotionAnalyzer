# 프로젝트 요약: PyVista 애니메이션 뷰어 v1

## 1. 완료된 작업

**주요 작업 파일:** `main_visualizer_v1.py`

**구현된 주요 기능:**
*   **데이터 로딩**: PVD (ParaView Data) 컬렉션 파일 또는 VTP (VTK PolyData) 파일을 로드하며, "box"와 "marker" VTP 파일을 구분합니다. PVD 파일을 찾지 못하거나 파싱에 실패할 경우, glob 패턴을 사용하여 파일을 검색하는 대체 기능을 포함합니다.
*   **시각화**:
    *   3D 박스 및 마커 메시를 표시합니다.
    *   박스 모서리 및 마커에 대한 라벨을 시각화합니다.
    *   참조를 위한 바닥 평면을 포함합니다.
*   **애니메이션**:
    *   키보드 입력('m' 키: 다음 프레임, 'n' 키: 이전 프레임)으로 프레임 단위 애니메이션을 제어합니다.
    *   프레임마다 메시와 라벨을 동적으로 업데이트합니다.
*   **사용자 상호작용**:
    *   시점 변경 ('1' 키: XY 평면, '2' 키: YZ 평면, '3' 키: ZX 평면).
    *   'q' 키를 통한 정상적인 프로그램 종료 (`sys.exit()`).

**적용된 주요 기술 및 개선점:**
*   **플로터**: 렌더링을 위해 `pyvista.Plotter`를 사용합니다.
*   **라벨 업데이트**: 안정적인 라벨 업데이트 메커니즘을 구현했습니다:
    *   라벨을 전용 `pyvista.PolyData` 객체(`box_label_polydata`, `marker_label_polydata`)와 연결했습니다.
    *   매 프레임마다 이 `PolyData` 객체의 `.points` 속성을 새로운 좌표로 **직접 수정(in-place modification)**했습니다.
    *   가장 중요하게, 포인트 업데이트 후 `PolyData_object.Modified()` (첫 글자 대문자 'M')를 호출하여 PyVista 파이프라인에 변경 사항을 알렸습니다. 이것이 핵심적인 수정 사항이었습니다.
*   **래핑 함수**: `PolyData` 포인트 업데이트 및 `.Modified()` 호출 로직을 캡슐화하는 `update_polydata_points` 함수를 도입하여 코드 구조, 가독성 및 안정성을 향상시켰습니다. 이 함수는 기본적인 타입/유효성 검사를 포함하며, 다양한 포인트 수에 대한 경우(예: 남는 라벨 숨기기)를 처리합니다.
*   **타입 힌트**: 최신 Python 3.9+ 스타일로 타입 힌트를 현대화했습니다:
    *   제네릭 타입에 `list`, `tuple`과 같은 내장 타입을 사용했습니다 (예: `list[str]`).
    *   Optional 타입을 위해 `| None`을 사용했습니다 (예: `pv.Actor | None`).
    *   콜백 함수 시그니처에 `collections.abc.Callable`을 사용했습니다 (예: `Callable[[int], None]`).
    *   `typing` 모듈에서 불필요한 임포트(`List`, `Tuple`, `Optional` 등)를 제거했습니다.
*   **설정**: `config.py`를 통한 기본적인 외부 설정을 사용합니다 (독립 실행 테스트를 위해 스크립트 내에 더미 버전도 포함).
*   **오류 처리**: 파일 로딩 및 프레임 업데이트 중 발생할 수 있는 오류 및 경고에 대한 `print` 문을 추가했습니다. `config.py` 또는 데이터 파일이 없을 경우 테스트를 위한 더미 VTP/PVD 파일 생성 로직을 포함합니다.

## 2. 개발 과정에서의 주요 결정사항 및 주의점

*   **라벨 업데이트 메커니즘 (가장 중요한 수정)**:
    *   초기에 `PolyData` 객체의 `.points` 속성에 *새로운* NumPy 배열 객체를 할당하는 방식(예: `polydata.points = new_np_array`)은 `.Modified()`를 호출하더라도 라벨의 시각적 업데이트를 유발하지 못했습니다.
    *   **성공적인 접근 방식**은 `polydata.points`가 참조하는 **기존** NumPy 배열의 *내용*을 직접 수정하고(in-place modification, 예: `polydata.points[i,:] = ...`), 그 후 `.Modified()`를 호출하는 것이었습니다. 이는 라벨 시스템이 의존하는 것으로 보이는 참조 무결성을 유지합니다.
*   **`.Modified()` 대소문자**: 정확한 PyVista (및 VTK) 메서드는 `ObjectName.Modified()` (첫 글자 대문자 'M')입니다. 이를 일관되게 사용하는 것이 중요했습니다.
*   **라벨용 `PolyData`**: 라벨은 점 위치를 정의하는 `PolyData` 객체에 연결됩니다. 이 `PolyData` 객체들은 `config.py`에 정의된 라벨 수만큼의 점을 가지도록 초기화됩니다. 만약 VTP 프레임이 정의된 라벨 수보다 적은 점을 가지면, 남는 라벨들은 해당 `PolyData` 객체 내의 해당 점들을 멀리 떨어진 위치로 이동시켜 "숨김" 처리합니다. VTP 프레임이 더 많은 점을 가지면, 정의된 라벨 수에 해당하는 부분집합만 사용됩니다.
*   **`config.py` 의존성**: 스크립트는 경로 및 라벨 정의를 위해 `config.py` 파일에 의존합니다. 실제 데이터 없이 테스트하기 위해 `if __name__ == '__main__':` 블록 내에 `DummyConfig` 클래스와 더미 파일 생성 로직이 추가되었습니다.
*   **PyVista 버전**: 작업은 PyVista 버전 0.45.2를 대상으로 수행되었습니다.
*   **프로그램 종료**: 'q' 키의 동작을 `plotter.close()`에서 `sys.exit()`를 호출하는 함수로 변경하여 Python 스크립트가 완전히 종료되도록 했습니다.

## 3. 해결된 주요 문제

*   **라벨 업데이트 실패**: 애니메이션 시 라벨이 메시와 함께 업데이트되지 않던 문제. 이는 `PolyData.points`의 in-place 수정 방식과 정확한 `.Modified()` 사용을 통해 해결되었습니다.
*   **프로그램 비정상 종료**: 'q' 키가 PyVista 창만 닫고 스크립트는 종료되지 않던 문제. `sys.exit()`를 사용하여 해결했습니다.
*   **`modified()` 대소문자 오류**: 일관되게 `Modified()`로 수정했습니다.
*   **타입 힌트 스타일**: 최신 Python 3.9+ 표준으로 업데이트하여 명확성을 높이고 `typing` 모듈 의존성을 줄였습니다.

## 4. 향후 논의된 개선 아이템

*   **클래스 기반 리팩토링**: 현재 함수 기반 스크립트를 클래스(예: `AnimationViewer`) 기반으로 변환하여 상태(플로터, 로드된 데이터, 프레임 인덱스 등)를 더 잘 관리하고 기능을 캡슐화합니다. 이는 코드 구성, 유지보수성 및 확장성을 향상시킬 것입니다.
*   **GUI 통합**:
    *   PyQt5 또는 PySide2와 같은 라이브러리를 사용하여 PyVista 플로터를 GUI 애플리케이션에 임베드합니다.
    *   파일 열기 대화상자, 설정 패널, 제어 버튼 등 UI 요소를 추가합니다.
*   **애니메이션 제어 기능 강화**:
    *   직접적인 프레임 탐색을 위한 **스크롤바/슬라이더**를 구현합니다.
    *   애니메이션 제어를 위한 **재생/일시정지/정지 버튼**을 추가합니다 (재생에는 `plotter.add_timer_event` 사용 가능성).
*   **더 유연한 설정 관리**: `config.py` 외에 또는 추가적으로 GUI를 통해 설정을 변경할 수 있도록 합니다.
*   **`update_polydata_points` 개선**: 만약 메시가 프레임 간 토폴로지(점의 수나 연결 구조)가 크게 변경될 수 있다면, `update_polydata_points`에서 동적 포인트 수 변경을 처리하는 방식을 더 개선합니다 (예: `PolyData`를 새로 만들고 라벨을 다시 추가. 단, 이는 깜빡임 문제를 다시 야기할 수 있음). 현재 방식은 `config.py`에 의해 라벨 수가 고정된다고 가정합니다.
*   **오류/경고 로깅**: 오류/경고에 대한 `print()` 문을 더 견고한 로깅 메커니즘으로 대체합니다.

이 요약이 향후 프로젝트 작업에 좋은 참고 자료가 되기를 바랍니다.
